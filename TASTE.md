# TASTE — 내 판단·문장 기준

이건 참고 문서가 아니라 **행동 기준**이다. 에이전트는 매 작업에서 이 문서대로 판단하고 쓴다.
일반적인 "좋은 글/좋은 설계"와 충돌하면 이 문서가 이긴다.
글을 쓸 때: 추상어·격언·자기칭찬을 먼저 지우고, 상황→판단→근거 순으로 다시 쓴다.
판단할 때: "이게 다음 결정을 바꾸나"를 통과 못 하면 버린다.
나에게 이 문서는 거울이다 — 기준이 흔들릴 때 여기로 돌아온다.

---

## 0. 적재·압축 규칙 (이 문서를 디벨롭하는 법)

- **적재 시점**: workflow-master 종료 시(`[WORKFLOW:LEARNING]`), daily-feedback 회고에서 **사용자가 직접 수정했거나 콕 집어 평가했을 때만**, 고찰 세션. 무확인 통과("이대로 저장" 클릭만)는 선호 신호가 아니므로 적재 금지.
- **적재 게이트**: 추가 전에 분류한다 — 횡단 판단 원칙이면 §1, 글 톤이면 §2, 특정 도구 함정이면 §3. 분류가 안 되면 §3. **단, PA·SharePoint·MS365 함정은 여기 말고 power-automate 스킬 `references/lessons-learned.md`로** (PA 작업 시만 로드).
- **단일 템플릿** (§1·§3 공통, 형식 발산 금지):
  ```
  ### {한 줄 교훈 — 명령형 판단문} #태그
  - 교훈: 무슨 일이 났나 / 무슨 규칙인가 (1-2줄)
  - 적용: 다음에 무엇을 하나 (1줄, 행동 가능하게) (출처: workflow_YYYY-MM-DD_{주제}.log)
  ```
- **중복 우선 병합**: 유사 항목이 이미 있으면 새로 만들지 말고 기존 항목을 갱신한다.
- **상한**: §2는 섹션(Headline/Cut/Core/Logic)당 Good 3 + Bad 3. 초과 시 가장 오래되고 최근 패턴과 중복인 사례를 §4로 졸업.
- **압축 트리거**: 적재 직후 `wc -l`로 줄 수 확인 — **600줄 초과면 같은 세션에서 압축을 제안**하고, 900줄이면 강제 압축. 이 파일은 매 세션 상시 로드되므로 길이가 곧 비용이다.
- **졸업**: 압축 세션에서 항목마다 "지난 분기에 이게 내 행동을 바꿨나"를 판정 — 못 바꾼 항목은 §4로 이동(삭제 아님, 재승격 가능).
- **append는 각 섹션 끝에만** (중간 삽입 금지 — 다중 세션 git 충돌 위생).

## 1. 판단 원칙

### 측정의 메타 기준 — "다음 결정을 바꾸는가" #Measurement
- 교훈: 품질을 재는 절대 기준은 없다. 다음 행동을 못 바꾸는 지표는 전부 허영(vanity)이다. (산출물 수·리뷰 건수가 정확히 그랬다)
- 적용: 지표·산출물 검증 시 4게이트 — ①행동 가능성(나쁘면 뭘 바꿀지 즉시 떠오르나) ②반증 가능성 ③조작 저항성(Goodhart) ④비용 대비. 품질은 정확성·적합성·단순성·지속성·비용으로 분리해 보고, 늦게 드러나는 단순성·지속성은 지연 측정(N일 뒤 살아남았나)으로 남긴다. (출처: 고찰 2026-05-31)

### 만들기 전에 멈춘다 — 기존 도구·제약 먼저 #Architecture
- 교훈: "입력 소스 통합" 요구에 4역할 토론+개발 계획까지 갔다가, 기존 도구(Notion/Obsidian)로 80% 커버 가능 결론. 사내 자동화도 NestJS 설계 후 "MS 도구만" 제약으로 전면 전환.
- 적용: 신규 제안 전 ①환경 제약(라이선스 등)부터 확인 ②"기존 도구로 어디까지 되나" 리서치 ③기존 도구 1-2주 사용 → 마찰 포인트가 곧 MVP 우선순위. 코딩 없는 접근이 유지보수 비용도 낮다. (출처: universal-inbox, security-automation)

### Less is More — 빼는 것도 개선 #L-0320-1
- 교훈: 담당자 "메일은 안 본다" → 메일 발송 제거, 흐름이 오히려 깔끔해짐.
- 적용: 자동화/제품 개선 = 추가만이 아니라 안 쓰는 기능 제거 포함. 사용자가 실제로 보는 채널에 집중.

### 교체 임계 — 너무 다르면 버리고 새로 짓는다 #L-0417-1
- 교훈: "동작이 기대와 너무 달라서 새로 하는 게 낫다" — koco-tester-manage를 버리고 재작성.
- 적용: 진단 비용이 재작성 비용을 넘는 지점이 있다. 그땐 고치지 말고 버린다.

### 차단 시 기존 합의를 뒤집는 대안은 자율 결정하지 않는다 #L-0611-1
- 교훈: KOCO SDK 라이브 작업 중 차단 우회 대안이 "CDN 미사용" 합의를 위반 — 사용자가 잡아 더 단순한 해법으로 회귀. (출처: 2026-06-11 회고)
- 적용: 합의 기록을 결정 전에 재확인하고, 뒤집어야 하면 묻는다.

### 한도는 쿼터와 같은 단위로 설계한다 #L-0611-2
- 적용: 비용이면 비용, 건수면 건수 — 단위 정합이 먼저, 숫자는 그다음. 검사 리포트의 건수도 단위를 명시한다 (보안 린트 "69건"=자산 ID 기준 vs 셀 121칸 — 단위 미명시로 후속 집계가 어긋남, 2026-06-12). (출처: 2026-06-11 KOCO SDK 회고)

### 기준 완화는 전제가 책임지는 범위까지만 #L-0610-1
- 적용: 인증/인프라 전제로 완화할 수 있는 건 그 전제가 커버하는 범위뿐 — 인가·무결성·가용성은 전제와 무관하므로 완화하지 않는다. (출처: 2026-06-10 회고)

### LLM 산출물 품질은 재현성 실험으로 판정한다 #L-0610-2
- 교훈: "잘 만들어진 것 같은데"를 감으로 넘기지 않고 재생성 재현성으로 판정. 슬러그 Jaccard는 개념 분산과 표기 분산을 구분해 해석할 것.
- 적용: 스키마 개선은 분산 지점(run마다 갈리는 곳)보다 공통 결함(모든 run이 똑같이 틀리는 곳)부터 — "수치 보존 규칙" 1줄이 보조 커버리지 76~86%→100%. (출처: vertical-wiki 2026-06-10)

### 쓰기격리 4원칙 — 쓰기/삭제 전 게이트 질문 #쓰기격리
어떤 도구든 쓰기 권한을 다룰 때의 횡단 원칙. 같은 원칙 위반이 3건 쌓이면 문구·우선순위를 재검토한다.

| # | 원칙 | 한 줄 | 게이트 질문 (Y/N) |
|---|------|-------|-------------------|
| P1 | 권한 비대칭 | 읽기는 넓게, 쓰기는 좁게 | read-only 자격증명으로 시작, 쓰기는 별도 발급 + 최소 스코프인가? |
| P2 | 격리 후 승격 | 본판에 직접 쓰지 않는다 | branch/plan/transaction/dry-run/staging에서 먼저 쓰는가? |
| P3 | 가역성 | 쓰기 전에 되돌릴 길부터 판다 | 롤백 수단(백업/down migration/soft delete)이 변경 **전에** 확보됐는가? |
| P4 | 단일 게이트 | 쓰기 경로는 하나, 우회로 금지 | 사람·에이전트 모두 같은 리뷰/승인 경로를 통과하는가? |

적용 예: DB 기본 DSN은 read-only 유저, 쓰기는 별도 유저+최소 스코프 / git은 branch→PR(main 직접 push 금지), Terraform은 plan까지·apply는 승인 후, k8s는 `--dry-run=server`→staging→prod / append-only·soft delete 우선, 위험 작업 직전 스냅샷, down migration 없는 DDL 머지 금지 / 스키마·인프라 변경은 PR이 유일 경로, 에이전트도 같은 게이트 통과("에이전트라서 자동 머지" 금지).

위반 로그: 쓰기 사고 시 `### [Pn 위반] 요약 / 위반 1줄 / 적용 1줄` 형식으로 여기 아래에 1분 안에 기록. (현재 기록 없음)

### 이론적 오탐 우려를 1회 실행으로 기각하지 않는다 #측정설계
- 교훈: 린트 설계 시 "vault 자동 커밋이 git 날짜를 오염시킨다"는 우려를 첫 실행 결과(위반 1건뿐)로 기각했으나, 그 1건이 바로 NFC 경로 복원 재추가 오탐이었다.
- 적용: git 기반 신선도 비교는 `--diff-filter=M`(실제 수정만)이 기본값. (출처: 2026-06-12 보안 대장 린트)

### 상태 변화 dedup은 최위험 단계에서 예외를 둔다 #알림설계
- 교훈: "상태 전이시 1회 발송"만 두면 중계 1회 실패가 영구 누락이 된다 — 소음 억제는 여유 단계(D-14/D-3)에서만 지키고, D-0/만료는 매 실행 발송이 안전장치다.
- 적용: 알림 dedup 설계 시 "전달이 한 번 실패해도 다시 울리는가"를 게이트 질문으로. 침묵이 기본인 자동화는 heartbeat(state mtime)로 죽은 침묵을 구분하고, 해소(RESOLVED) 1회 통보로 조치 확인 루프를 닫는다. (출처: workflow_2026-06-12_credential-expiry-audit.log)

### 지연 측정의 회수 트리거를 "직전·연속성"에 무임승차시키지 않는다 #측정설계
- 교훈: workflow FATE 회수가 "직전 로그"만 보게 설계돼 주제가 점프하면 PENDING이 영구 고아가 됐다(94건 중 회수 27건). INCOMPLETE 스캔도 오탐 5/6이라 alert fatigue로 진짜 1건마저 묻혔다.
- 적용: 회수 대상은 미회수 잔액 전체(오탐 필터 후), 라이브 검증 통과분은 즉시 종결해 큐에 안 넣는다. 스캔은 workflow-audit.sh가 세션 시작마다 자동 실행. (출처: workflow_2026-06-12_workflow-meta-review.log)

## 2. 글 톤 & 회고 패턴

### 다섯 규칙 (모든 글·회고에 적용)

1. **내 실제 발화는 따옴표로 그대로 살린다.** 멋진 표현으로 바꾸지 마라. ("동작이 기대와 너무 달라서 새로 하는 게 낫다" 채택 / 같은 내용을 "손익분기점"으로 추상화한 안은 거부)
2. **추상어·격언·경영서 톤 금지.** "손익분기점·레버리지·기준·전략", "완벽히 준비된 적은 없었다"류 자기계발서 문구가 나오면 구체 사건·라벨로 치환. 태그도 "Antifragile" 말고 "Less is More, Fail-fast" 같은 구체 라벨.
3. **자기칭찬·속도자랑·완수자랑 금지.** "표면 증상에 속지 않고 원인까지 추적"(자기칭찬), "당일 반영 완료, 빠르게 전환"(속도자랑)은 회고가 아니다. 예외: 구축→배포를 하루에 닫은 "단일 완결 서사"는 나열이 아니면 그날의 의미로 인정한다.
4. **단위는 상황→판단→근거.** "~하고 ~하고 ~함" 작업 나열 금지. 여러 축이면 한 축으로 좁힌다.
5. **과잉 해석 금지.** 관찰과 판단까지만 쓴다. 파생 리스크·일반화는 본인 확인 전에 붙이지 않는다. ("헬퍼 미러링 재발 위험"까지 덧붙였다가 "무슨말인지 모르겠음" 반려)

### 회고 메타 규칙

- 같은 교훈 반복 시 Logic을 새로 만들지 않는다 — Cut에서 "왜 이미 배운 걸 또 못 지켰는지" 파기
- 억지로 Logic을 만들지 않는다. 없으면 "새로운 Logic 없음"
- 기술 디테일은 회고에 넣지 않는다. 회고는 상황-판단-결과의 맥락
- 체크리스트성 항목은 Logic이 아니다 — 릴리즈 체크리스트 등 해당 문서로 보낸다 (L-0611-3 사례)
- 한 실험에서 나온 묶음 배움은 분리를 강제하지 않는다 (L-0610-1·2 "묶어서 유지" 선택)
- 글쓰기 원칙을 새로 받으면 전면 적용 전에 before/after 샘플 1개로 강도를 캘리브레이션한다 — 원칙의 적용 범위를 좁히는 게 톤을 지키는 길 (daily-news 4차)
- 무확인 통과는 선호가 아니다 — "이대로 저장"만으로 Good 적재 금지. 사용자가 고치거나 콕 집어 평가한 것만 적재한다 (2026-06-12, "확인 안 하고 넘긴 것도 선호로 남았다" 지적)
- 자동 회고의 주체 구분 — `오늘의 중심 질문`·`내일 의도`·`내일 첫 액션`·Spaced는 사용자가 직접 말한 관심·불편·판단에서만 뽑는다. 에이전트가 대신 한 크론 수정·문서 정리·후보 필터링은 `운영 처리`로 분리하고 사용자 의도로 승격하지 않는다 (2026-06-10, SAM 운영 작업이 `내일 의도`로 올라간 지적)
- 회고 본문도 골격을 회전한다 — `오늘 마스터님이 붙잡은 질문은 ~였습니다` 고정 오프닝, `X가 아니라 Y` 대비 프레임의 연일 반복 금지. 아래 Threads 문체의 골격 회전 규칙을 회고 본문에도 준용 (2026-06-12)

### Headline 패턴

- Good: "담당자가 '메일 필요 없다'고 말한 순간 — 기능을 빼는 것이 더 나은 자동화라는 판단" (상황+판단+의미)
- Good: "'동작이 기대와 너무 달라서 새로 하는 게 낫다' — koco-tester-manage를 버리고 invitation을 다시 지은 날" (한 축 집중, 발화 인용)
- Good: "머릿속 노하우를 위키로 자산화하면서, '잘 만들어진 것 같은데'를 감으로 넘기지 않고 재현성 실험으로 판정한 날" (판단 축 선명)
- Bad: "법무 자동화 2개 피드백 반영 완료한 날 — 이력관리 워크플로우도 스킬로 정착" (작업 완료 나열)
- Bad: "ZeroTier로 사내망 뚫기 성공 — 도구가 다르면 방화벽도 다르게 반응한다" (기술 작업 로그)
- Bad: 초안 "차단마다 '기존 합의'와 '단위 정합'을 기준으로 되돌아본 날" → 최종은 실제 사건·발화 인용으로 교체됨 ("기준"류 추상어는 경영서 톤)

### Cut 패턴

- Good: "L-0317-1을 알면서도 가품 캡쳐에서 또 탐색부터 시작. 새 기능이라는 흥미가 검증 단계를 건너뛰게 만드는 패턴" (미실행 원인을 행동 패턴으로)

### Threads 문체 (THREADS_STYLE_HISTORY.md 병합, 2026-06-12)

- 확정 톤 (03-13·03-31 승인): 담백·짧고·현실적인 줄글, 3~5줄에 핵심. 긴 주제는 메인 포스트 + 첫 댓글 직접 달기. 과장된 결론 대신 상황에 따라 달라지는 판단.
- **골격 회전 — 승인 샘플에서 톤만 가져간다 (05-11·06-12 두 번 지적)**: "처음에 한번 그 상황에만 맞았던 패턴"을 다시 쓰지 않는다. 작성 전 최근 7개 후보의 첫 줄·마무리 구조를 확인하고 겹치면 폐기. 기본값 금지 골격: 첫 줄 `오늘은 X보다 Y`, `오늘 배운 것:` 오프닝, 격언화 마무리(`결국 ~인 듯`, `~것 같다`, `~에 가깝다`, `~할 때가 있다`), `주장 → 오늘 사례 → 일반화 교훈` 3단 고정.
- 각도를 매번 새로 잡는다: 사건형/판단형/반성형/관찰형/대화형/숫자형 중 2개 이상 다른 각도로 후보를 낸다. 기본형/라이트형은 같은 글의 길이 차이가 아니라 각도가 다른 두 글.
- 첫 줄은 주장/반전/선택 압력 (06-03): "어디에서 살아날 언어인가" 같은 추상 요약 금지. 데이터 근거가 있는 날은 구체 대비를 먼저 — "네이버는 `신혼여행`을 밀어줬고, 유튜브는 `세계일주`를 밀어줬다."
- Threads 후보는 회고 요약의 재포장이 아니라 독립된 짧은 글. 말미의 인용되기 쉬운 한 줄도 격언이 아니라 구체 문장으로.
- 앱 제작 공유(World Travel)는 경량 캡션 포맷 (06-04): `D+037/180`, `오늘 본 거 / 고친 거`. 장식형 진행도 시안은 "완전 별로" 반려 — `D+ 한 줄 + 60점 맵`. 단 캡션 포맷이 본문 골격 면허는 아니다.

### #문서화

- **1인칭 섹션 제목 금지** — 소유권은 표 컬럼(역할: 주도/팀)으로.
- **살아있는 자원 목록엔 "목록 기준일" + 정답 출처 링크**를 함께 둔다.
- **Obsidian에 HTML `<details>` 금지** — foldable callout(`> [!example]- 제목`) 사용.
- **TODO 마커는 원본 셀에만** — `[미입력]` 같은 린트 집계 토큰을 파생 요약 표에 재인용하면 이중 집계된다(69→84건 오염). 파생 표는 그룹 제목으로 상태를 표현. (2026-06-12 credential targets)

### #skill

- **레포 스킬의 글로벌 전환 시**: cwd 상대 경로를 스킬 디렉토리 기준 절대 경로로 고정, "잘 돌던" 스크립트의 숨은 .venv 의존을 실측. (daily-news 2026-06-12)

### #리스크관리

- **"문제 없음"은 해소인지 오탐인지 구분해 확인** — 대장 표기가 다르다 (해소=이력 유지, 오탐=목록 제거).

### #git

- **충돌 해소는 "덮기"가 아니라 "병합"** — 해소 직후 양쪽 원본 대비 full diff로 "의도한 차이만 있는지" 확인. (Infinity 섹션 유실 사례 2026-06-12)

### #cmux

- **cmux CLI는 에러여도 exit 0**: 모든 에러가 exit 0 + 출력의 `Error:`/`not_found:` 텍스트로만 옴 — `$?` 분기 무용. `ping`→`PONG` 진입 게이트 + 문자열 매칭으로 분기. `read-screen`은 위치인자 불가, `--surface` 플래그 필수. (meta-orchestrator 2026-06-12)
- **`cmux notify`는 OS 배너로 안 뜬다**: OK 반환하지만 list-notifications에도 안 남고 사용자에게 미도달 — 사용자 푸시는 `osascript display notification`(실측 도달), 보조로 `trigger-flash`/`afplay`. 프롬프트 줄(`❯`)의 텍스트는 미제출 입력이 아니라 미리보기일 수 있음 — send로 새 입력을 보내 판별. (meta-orchestrator 2026-06-12)
