# [wiki-02] Content Navigation Fix

- timestamp: 2026-04-19T03:07
- trigger: 사용자 "wiki가 인덱스 페이지만 있고 실제 노트는 안보여"
- scope: 02:45 초기 배포 후 실제 콘텐츠(diary, sources, mapped) 접근 불가 이슈 수정

## 진단

Pages source `/docs`로 활성화했으나 agent-wiki 레포의 실제 콘텐츠(`diary/`, `concepts/`, `mapped/`, `sources/`, `syntheses/`, `maintenance/`)는 모두 루트에 존재. Pages가 `/docs` 폴더만 서빙해서 Docsify가 이들을 fetch할 수 없었음.

## 조치

1. `docs/` 폴더 제거, `index.html`을 루트로 이동 (commit b7a8d27)
2. Pages source `/docs` → `/` 변경 (`gh api PUT /repos/.../pages`)
3. 과도기 빌드 1회 실패 → `gh api POST /pages/builds`로 재트리거
4. `.nojekyll` 추가 (commit b255a48) — Jekyll이 `.md`를 `.html`로 변환하고 `_sidebar.md`의 `_` 접두사를 배제해서 Docsify가 raw markdown fetch 실패
5. `_sidebar.md` 확장 (commit ba2f28a) — sources(7개 카테고리) + mapped(10개 폴더 대표 노트) 추가

## 최종 검증 (ba2f28a, HTTP 200 확인)

- `/` (Docsify 진입) → 200
- `/_sidebar.md` → 200
- `/diary/2026-04-19.md` → 200
- `/sources/Fundamental.md` → 200
- `/mapped/Fundamental/Architecture.md` → 200
- `/README.md` → 200

## Key Lesson

**Docsify + GitHub Pages는 `.nojekyll` 없으면 동작하지 않는다.** Jekyll이 기본으로 활성화되어 있어 `.md`를 `.html`로 변환하고 언더스코어 파일(`_sidebar.md`, `_navbar.md`)을 배제함. 설계안과 초기 구현에 이 핵심 디테일이 빠져 있어서, 배포는 "성공"하지만 실제 콘텐츠 탐색은 완전히 깨진 상태였음.
