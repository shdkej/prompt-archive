# OpenClaw 채팅방 에이전트 설계 패턴 리서치 보고서

## 1. OpenClaw 개요

OpenClaw는 2025년 말 공개된 오픈소스 멀티에이전트 프레임워크로, 2026년 2월 GitHub 스타 100,000개를 돌파한 바이럴 도구입니다. "Space Lobster" 컨셉의 개인 AI 어시스턴트로, 로컬 또는 클라우드에서 실행하며 앱과 온라인 생활 전반에 걸쳐 자율적으로 작업을 처리합니다.

### 핵심 특징
- **하나의 Gateway 프로세스**에서 여러 에이전트를 실행
- 각 에이전트는 독립적인 파일, 메모리, 인증, 툴을 보유
- WhatsApp, Telegram, Slack, Discord와 연동 가능 → **채팅방 친화적**

---

## 2. OpenClaw의 기본 아키텍처: 메시(Mesh) 토폴로지

OpenClaw는 기본적으로 **메시(Mesh) 토폴로지**를 채택합니다.

| 항목 | 내용 |
|------|------|
| 통신 방식 | 에이전트 간 P2P (중앙 코디네이터 불필요) |
| 장점 | 복잡하고 상호의존적인 작업 모델링에 적합 |
| 병렬성 | 여러 에이전트가 서로 정보를 교환 후 결과 생성 가능 |

**Paperclip(경쟁 프레임워크)**은 계층적 Supervisor-Worker 아키텍처를 사용하는 반면, OpenClaw는 메시 토폴로지가 기본값입니다. 즉, 리더+병렬 실행자 구조는 OpenClaw의 기본 설계가 아니라 **선택적으로 구현하는 패턴**입니다.

---

## 3. Lobster 워크플로우 엔진

Lobster는 OpenClaw 네이티브 워크플로우 쉘로, 스킬/툴을 구성 가능한 파이프라인과 안전한 자동화로 전환합니다.

### 핵심 역할
- **결정론적 파이프라인**: 여러 번의 tool call 왕복 → 단일 호출로 대체
- 에이전트가 "무엇을 어떤 순서로, 어떤 입력으로, 어떤 조건에서" 실행할지 정의
- 승인 체크포인트(human-in-the-loop) 지원
- 파이프라인이 데이터 형태 → 로깅, diff, 재현, 리뷰 용이

### Sub-lobster (중첩 워크플로우)
- `.lobster` 파일을 스텝으로 임베드 가능
- 조건부 루프 지원
- **클래식 Supervisor-Worker 패턴 구현에 적합**

### 채팅방 에이전트에서 Lobster 활용
```
채팅 메시지 수신
    ↓
Lobster 파이프라인 트리거
    ↓
[병렬 스텝] 의도 분류 | 감성 분석 | 컨텍스트 검색
    ↓
응답 생성 에이전트
    ↓
후처리 & 발송
```

## 4. 리더+병렬 실행자 패턴 — 필요한가?

### 패턴 설명
```
[리더 에이전트]
    ├── [실행자 A] 태스크 분류
    ├── [실행자 B] 데이터 검색
    └── [실행자 C] 응답 생성
```

### OpenClaw에서 구현 방법
- **오케스트레이터 에이전트** → 메인 채팅 처리 + 서브에이전트 스폰
- 서브에이전트는 백그라운드에서 독립 실행 → 결과를 메인에 반환
- `maxConcurrentRuns` 설정 + lane/queue 시스템으로 세션 격리

### 리더+병렬 실행자 패턴이 필요한 경우 ✅

| 상황 | 이유 |
|------|------|
| 요청당 다수의 독립적 서브태스크 존재 | 병렬화로 레이턴시 감소 |
| 명확한 작업 분해 가능 | 리더가 분해·위임·취합 |
| 품질 검토가 중요 | 리더가 최종 검수 |
| 비즈니스 워크플로우 추적 필요 | 감사 로그, 디버그 용이 |

### 이 패턴을 피해야 하는 경우 ⚠️

| 상황 | 이유 |
|------|------|
| 모든 결정이 리더를 통해야 함 | 리더가 병목(모놀리스 재현) |
| 에이전트 수 증가 | 단일 장애점(Single Point of Failure) |
| 실시간 채팅에서 초저지연 필요 | 리더 레이턴시 누적 |

## 5. 대안 패턴 비교

### A. 이벤트 기반(Event-Driven) / P2P 메시
```
에이전트 A ←→ 에이전트 B
    ↕               ↕
에이전트 C ←→ 에이전트 D
```
- OpenClaw 기본값
- 에이전트가 이벤트를 emit하고 listen
- **장점**: 높은 확장성, 리더 병목 없음, 실시간 반응성
- **단점**: 디버그 복잡, 흐름 추적 어려움

### B. 공유 블랙보드(Shared Blackboard)
- 에이전트들이 공유 메모리/상태를 읽고 씀
- 채팅방의 컨텍스트 공유에 적합
- OpenClaw의 shared memory 기능 활용 가능

### C. ClawTeam (자기조직화)
- 에이전트들이 스스로 팀을 구성
- 작업 분담, 소통, 결과 수렴을 사람의 micromanagement 없이 수행
- **채팅방에서 동적 역할 할당에 적합**

---

## 6. 채팅방 에이전트 설계 권장사항

### OpenClaw 채팅방 에이전트 설계 시

1. **기본은 메시 토폴로지** (OpenClaw 기본값 활용)
   - 단순한 채팅방 → P2P 에이전트 통신으로 충분

2. **복잡한 워크플로우는 Lobster 파이프라인 사용**
   - 결정론적 처리 필요 시 (예: 승인 흐름, 다단계 검증)
   - 리더 역할을 Lobster가 대신하여 LLM 오케스트레이션 비용 절감

3. **리더+병렬 실행자 패턴 적용 기준**
   - 요청당 3개 이상의 독립적 서브태스크가 있을 때
   - 품질 검수가 비즈니스 요구사항일 때
   - 단순 채팅방에는 **과도한 복잡도** 유발 가능

4. **실용적 조합 권장**
   ```
   채팅 수신 → [Lobster 파이프라인]
       ↓ 간단한 요청
   단일 에이전트 직접 처리
       ↓ 복잡한 요청
   오케스트레이터가 서브에이전트 병렬 스폰
   ```

---

## 7. 결론

**리더+병렬 실행자 구조가 채팅방 에이전트에 반드시 필요한 것은 아닙니다.**

- OpenClaw의 기본 메시 토폴로지로 대부분의 채팅방 시나리오 처리 가능
- 복잡한 워크플로우는 **Lobster 파이프라인**이 리더 역할을 대신하여 더 효율적
- 리더+병렬 실행자 패턴은 **명확한 작업 분해와 품질 검수가 필요한 복잡한 채팅방**에 선택적으로 적용
- 단순히 병렬화만 원한다면 OpenClaw sub-agents + `maxConcurrentRuns` 설정으로 충분
- 리더 병목 위험을 피하려면 **이벤트 기반 P2P** 또는 **ClawTeam 자기조직화** 패턴 고려

**권장 아키텍처**: OpenClaw 메시 + Lobster 파이프라인 (필요 시 오케스트레이터 서브에이전트 패턴 추가)
