**업로드 절차** (퇴근 처리 11번에서 참조):

> **주의**: `gog drive download`는 `-o` 플래그를 지원하지 않음. 기본 다운로드 경로: `~/Library/Application Support/gogcli/drive-downloads/<fileId>_<filename>`

1. `/tmp/diary-YYYY-MM.md`에 당일 리뷰 내용만 추출하여 임시 파일 생성
2. `gog drive ls --parent <폴더ID>`로 해당 월 파일(`YYYY-MM.md`) 존재 여부 확인
   - 있으면: `gog drive download <파일ID>` → 기본 경로에서 파일 읽기 → 당일 내용 추가하여 `/tmp/diary-YYYY-MM.md` 생성 → `gog drive delete <파일ID> --force` → `gog drive upload /tmp/diary-YYYY-MM.md --parent <폴더ID> --name YYYY-MM.md`
   - 없으면: 당일 내용으로 `/tmp/diary-YYYY-MM.md` 생성 → `gog drive upload /tmp/diary-YYYY-MM.md --parent <폴더ID> --name YYYY-MM.md`
3. 임시 파일 및 다운로드 파일 삭제

#### 2. 미디어 시청 기록 (자동 수집 — YouTube, Netflix 통합)

| 항목      | 값                                                              |
| --------- | --------------------------------------------------------------- |
| 소스      | Chrome 브라우저 히스토리 DB                                     |
| DB 경로   | `~/Library/Application Support/Google/Chrome/Profile 1/History` |
| 추출 방식 | DB 복사 후 SQLite 쿼리 (Chrome 잠금 회피)                       |

**추출 쿼리** (단일 쿼리로 모든 플랫폼 수집):

```sql
-- /tmp에 DB 복사 후 실행
SELECT
  CASE
    WHEN url LIKE '%youtube.com/watch%' THEN 'YouTube'
    WHEN url LIKE '%netflix.com/watch%' THEN 'Netflix'
  END AS platform,
  title, url,
  datetime(last_visit_time/1000000-11644473600, 'unixepoch', 'localtime') as visit_time
FROM urls
WHERE (url LIKE '%youtube.com/watch%' OR url LIKE '%netflix.com/watch%')
  AND date(last_visit_time/1000000-11644473600, 'unixepoch', 'localtime') = date('now', 'localtime')
ORDER BY last_visit_time DESC;
```

**처리 절차** (퇴근 처리 1번에서 참조):

1. 당일 시청 기록 추출 (플랫폼별 자동 분류)
2. 영상 제목으로 주제 분류 (학습/뉴스/엔터테인먼트 등)
3. Netflix title이 "넷플릭스"만 표시되는 경우, 같은 날 browse 페이지(`?jbv=`)의 title에서 작품명 매칭 시도. 알 수 없으면 "Netflix 시청"으로 기록
4. 데일리 로그의 "오늘 본 것" 섹션에 플랫폼별 그룹핑하여 추가

> 새 플랫폼 추가 시: CASE 절과 WHERE 절에 `url LIKE '%{도메인}%'` 조건만 추가

## 트리거

### 출근

**트리거**: "좋은 아침~"

**동작**:

1. 새 하루 업무 시작 인식
2. 어제 회고 검색 (두 소스를 확인, 어느 쪽이든 있으면 표시)
   - **Obsidian**: `~/Obsidian/diary/YYYY-MM.md`에서 어제 날짜(`## MM-DD`) 항목의 `#### 회고` 섹션 읽기
   - 없으면 생략 (에러 아님)
3. **미완료 TODO 및 리마인드 검색**: 최근 3일간 다이어리 항목에서 미체크 TODO(`- [ ]`)와 '기타' 섹션의 생활 메모(예: "컵 들고오기") 중 아직 유효한 항목을 리마인드
4. 이전 대화는 참고용으로만 (과도한 참조 금지)
5. 오늘 주요 업무/목표 확인

**응답 형식**:

```markdown
## 좋은 아침!

### 어제 회고 ({날짜})

- Cut: {반복된 행동/판단 패턴 — 있을 때만}
- Core: {어떤 상황에서 어떤 판단을 내렸는지}
- Logic: {삶에 남기는 규칙 — 있을 때만} `L-MMDD-순번`

### 리마인드

- {미완료 TODO 또는 기타 메모 중 유효한 항목}

### 오늘 계획

> 다이어리에 오늘 할 일이 이미 적혀있으면 아래 테이블로 정리하고, 없으면 "오늘 어떤 작업을 할 예정인가요?"로 물어본다.

| #   | 작업     | 완료 상태        | 예상 시간        |
| --- | -------- | ---------------- | ---------------- |
| 1   | {작업명} | {완료 기준 정의} | {예상 소요 시간} |

- 완료 상태: 각 작업의 "Done"이 무엇인지 구체적으로 정의
- 예상 시간: 이전 세션 기록과 작업 성격을 고려하여 산정
- 블로커가 될 수 있는 작업(외부 의존, 확인 작업)은 오전 우선 처리 제안
- 합계 시간도 표시하여 하루 분량 감각 제공
```

> 어제 회고 문서가 없으면 "어제 회고" 섹션은 생략하고 바로 오늘 목표를 물어본다.
> 리마인드 항목이 없으면 "리마인드" 섹션은 생략한다.

## 퇴근 처리

**세션 수집 스크립트**: `~/workspace/prompt-archive/scripts/daily-sessions.py`

**처리**:

1. **오프라인 소스 수집** — 각 소스의 처리 절차는 [§ 오프라인 데이터 소스](#오프라인-데이터-소스) 참조
   - Obsidian inbox 처리
   - Chrome 히스토리 미디어 시청 기록 추출
2. 스크립트 실행하여 세션 목록 수집
   - 오늘: `python3 ~/workspace/prompt-archive/scripts/daily-sessions.py`
   - 어제: `python3 ~/workspace/prompt-archive/scripts/daily-sessions.py {어제날짜}`
   - 특정일: `python3 ~/workspace/prompt-archive/scripts/daily-sessions.py YYYY-MM-DD`
3. 스크립트 출력(프로젝트명, 분류, 사용자 메시지)을 기반으로 각 세션의 핵심 작업 요약
4. 필요시 개별 세션 JSONL 파일을 읽어 상세 내용 보강
5. **다이어리 '기타', '인사이트' 섹션 수집**: 당일 Obsidian 다이어리의 '기타', '인사이트' 항목을 읽어 인사이트(업무 외 깨달음, 아이디어)와 리마인드(생활 메모, 미완료 TODO)로 분류. 인사이트는 `#### 인사이트` 섹션에, 생활 메모는 `#### 리마인드` 섹션에 반영
6. **결정 맥락 추출 (Decision Context Extraction)**
   - 세션 대화 + Obsidian inbox 메모에서 의사결정 패턴 탐지: "A vs B", "고민", "선택", "결정", "트레이드오프", "이유는", "왜냐면", "대신" 등
   - 탐지된 결정마다 Context/Options/Decision/Reasoning/Trade-off 구조로 정리
   - **업무/개인 분류**: 세션의 프로젝트 분류(`~/dev/` → 업무, `~/workspace/` → 개인)를 기준으로 결정도 자동 분류. inbox 메모는 내용으로 판단
   - 결정이 없는 루틴 작업 세션은 "새로운 결정 없음"으로 표기
7. **회고 추출 - Life Feedback Loop** — 품질 기준은 [§ 회고 품질 기준](#회고-품질-기준-goodbad-예시) 참조
   - AI가 세션 + 오프라인 소스를 분석하여 초안 자동 생성
   - 헤드라인 / Cut / Core / Logic 각 항목 작성
8. **Planner Agent 리뷰**: 회고 초안을 `planner` agent(`~/.claude/agents/planner.md`)에게 전달하여 품질 검증
   - Agent 호출: `Agent(subagent_type="planner", prompt=...)`
   - planner에게 전달하는 내용:
     - `~/workspace/prompt-archive/review-patterns.md` (사용자 톤/취향 패턴 — 반드시 먼저 읽기)
     - 회고 초안
     - 최근 7일 회고 (패턴 비교용)
   - planner는 패턴 파일의 Good/Bad 예시를 기준으로 초안을 검증하고, 개선된 초안을 반환
   - planner가 "충분히 좋다"고 판단하면 그대로 통과
9. **사용자 확인**: 결정 로그 + planner 리뷰 반영된 회고 + 오늘 본 영상을 함께 보여준 뒤, 하루 데이터 기반 맥락 질문 + 선택지를 제시
   - **맥락 질문 생성**: 세션 데이터/다이어리/로그를 분석하여 2~4개 질문을 자동 생성. 질문 유형:
     - **빠뜨린 결정**: 세션에서 "A vs B", 선택지 논의가 있었지만 결정 로그에 안 잡힌 것 (예: "사건사고에서 첨부파일 방식을 고민한 것 같은데, 결론이 났나요?")
     - **후속 맥락**: 완료된 작업의 배경이나 다음 단계 (예: "보안 패치 후 배포는 언제 예정인가요?")
     - **감정/에너지**: 하루 전체 흐름에서 느낀 점 (예: "13개 세션을 돌았는데, 컨텍스트 스위칭이 부담되진 않았나요?")
     - **연결 인사이트**: 서로 다른 세션 간 연결점 (예: "AI 도입 4단계 메모와 사건사고 유형 AI 판단 설계가 연결되는 것 같은데, 의도한 건가요?")
   - 질문은 Y/N 또는 한 줄 답변으로 충분하도록 구체적으로 작성
   - 형식:
     ```
     ### 오늘 하루 돌아보기

     Q1. {맥락 질문 1}
     Q2. {맥락 질문 2}
     Q3. {맥락 질문 3}

     ---
     0 - 패스 (이대로 저장)
     ```
   - 사용자가 질문에 답변하면 → 해당 내용을 결정 로그/회고/인사이트에 반영 후 저장
   - `0` 또는 "패스" → 초안 그대로 사용, planner 초안을 Good 패턴으로 `review-patterns.md`에 추가
   - 사용자가 회고 자체를 수정하고 싶으면 → 수정 전후 차이를 `review-patterns.md`에 기록 (원본 → Bad, 수정본 → Good, 왜 거부됐는지 추론 메모)
10. **Workflow Log Lesson-Learned 추출**: `~/.claude/logs/`에서 당일 로그 파일 수집 및 교훈 추출
    - 파일 패턴: `workflow_YYYY-MM-DD_*.log`, `troubleshooting_YYYY-MM-DD_*.log`
    - workflow 로그에서 `[WORKFLOW:LEARNING]` 섹션 추출
    - troubleshooting 로그에서 문제/원인/해결 패턴 추출
    - 추출된 교훈을 분류: **도구/프레임워크 TIL**, **프로세스 개선**, **실수 방지**
    - 교훈이 기존 CLAUDE.md의 Lesson Learn에 추가할 만한 것이면 별도 제안
    - 당일 로그 파일이 없으면 이 단계 생략
11. **Obsidian inbox.md 비우기**: [§ Obsidian 모바일 Inbox](#1-obsidian-모바일-inbox-수동-캡처) 처리 절차 4번 참조
12. **Google Drive 업로드**: 당일 리뷰 내용만 추출하여 [§ Google Drive 업로드 절차](#google-drive) 참조
13. 전체 요약 터미널 출력 (lesson-learned 포함)
14. 회고 작성 시 업무 외 항목도 가능하면 함께 반영 (영상 시청 경향/인사이트, 일상 리듬/생활 회고)

4. `~/.claude/project-categories.json`의 patterns로 업무/개인 분류

## 회고 품질 기준

> **핵심 원칙**: 기술 디테일이 아니라 **상황-판단-결과의 맥락**을 기록한다.
> **Good/Bad 예시 및 톤 패턴**: `~/workspace/prompt-archive/review-patterns.md` 참조 (planner 리뷰 시 반드시 읽기)

- **헤드라인**: 상황(갈등/도전) + 대응. 작업 나열 X
- **Cut**: 행동/판단의 반복 패턴. 기술 실수 나열 X. 없는 날 생략
- **Core**: 의사결정 전환점 (상황→판단→결과). 작업 후기 X
- **Logic**: 진짜 배운 것만. 매일 강제 생성 X. ID: `L-MMDD-순번`

## 오늘의 결정 (Decision Log) — 업무

> 무엇을 했는지보다 **어떤 고민 끝에 무엇을 결정했는지**가 미래의 나에게 더 가치 있다.

### 결정 1: {결정 제목} `#태그`

- **상황(Context)**: 어떤 문제/갈림길을 만났는가
- **선택지(Options)**: 어떤 대안들이 있었는가
- **결정(Decision)**: 무엇을 선택했는가
- **근거(Reasoning)**: 왜 이것을 선택했는가
- **트레이드오프(Trade-off)**: 이 결정으로 포기한 것, 감수한 리스크

## 예외 상황

- **기존 문서 존재**: 선택지 제시 (1: 업데이트 / 2: 새 문서 / 3: 취소)
- **연휴 후 복귀**: 빈 날짜 목록 제시하여 선택 (또는 "모두 넘어가기")
- **지연 정리**: 제목에 "(늦은 정리)" 표기, 작성일/업무일 명시
