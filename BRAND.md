# Sam Samuel 브랜드 아이덴티티

## 핵심 정보

| 항목       | 내용                                                    |
| ---------- | ------------------------------------------------------- |
| **이름**   | Sam Samuel (가칭 — 확정 전)                              |
| **슬로건** | 생산자를 만든다                                         |
| **미션**   | 시작의 장벽을 낮춰서, 만들고 싶은 사람이 만들 수 있게   |
| **비전**   | 모든 사용자가 생산자가 되는 세상                         |

## 왜 Sam Samuel인가

뭔가 만들고 싶은 마음은 있는데, 어디서부터 시작해야 할지 모르겠다. 도구는 넘치는데 첫 발을 떼기가 어렵다. Sam Samuel은 이 **시작의 장벽**을 문제로 본다.

우리가 해결하는 건 기술적 역량의 부족이 아니라, 시작하지 못하게 만드는 막연함이다. 복잡한 도구를 단순하게, 막막한 첫 화면을 친근하게 바꿔서, 사용자가 "일단 해볼까"라고 느끼게 만드는 것. 그게 Sam Samuel이 존재하는 이유다.

## 만드는 것

**사람들이 곤란을 겪고 있는 문제를 해결하는 모든 도구.**

특정 카테고리에 얽매이지 않는다. 누군가 막혀 있으면, 그 막힌 곳을 뚫어주는 도구를 만든다. 형태는 앱이 될 수도 있고, 자동화 워크플로우가 될 수도 있고, 단순한 유틸리티가 될 수도 있다.

판단 기준은 하나다: **이걸 쓰면 곤란이 해결되는가.**

## 대상

**생산 의지가 있는 모든 사람.** 직업, 규모, 경험 수준을 가리지 않는다.

- 사이드 프로젝트를 시작하고 싶은 직장인
- 아이디어는 있지만 구현 방법을 모르는 비개발자
- 새로운 도구를 배우며 성장하고 싶은 주니어
- 반복 업무를 자동화하고 싶은 1인 운영자

이들의 공통점은 하나다: **만들고 싶은 마음이 이미 있다.** Sam Samuel은 그 마음에 도구를 더해준다.

## 핵심 가치

세 가지는 동등하다. 어느 하나가 다른 것을 압도하지 않는다. 셋의 균형이 Sam Samuel다움이다.

1. **시작을 쉽게** — 첫 화면에서 뭘 해야 하는지 바로 알 수 있어야 한다
2. **스스로 성장하는 환경** — 가르치는 게 아니라, 써보면서 알게 되는 구조를 만든다
3. **최소한의 도구, 최대한의 가능성** — 기능을 쌓는 게 아니라 제약을 줄인다

## 편집 철학: 100에서 10을 남긴다

Sam Samuel의 모든 제품은 **'비움'이 아닌 '정제'**로 만들어진다.

0에서 10을 만드는 것은 심플함이 아니다. 100가지의 가능성과 기능을 깊이 이해한 후, 사용자의 삶에 가장 임팩트를 줄 수 있는 **본질적인 10%만을 남기는 것**이 우리의 편집이다.

| 구분 | 의미 |
| ---- | ---- |
| **휑함** | 아무것도 없는 곳에 적게 두는 것. 빈약함 |
| **심플함** | 가득 찬 곳에서 정교하게 덜어내어 핵심만 남긴 것. 우아함 |

- **해상도를 높이는 도구**: 사용자가 무심코 지나치던 문제나 데이터를 더 선명하고 직관적으로 마주하게 돕는 다리(Bridge) 역할
- **복잡성의 명료한 환원**: 내부가 아무리 복잡해도 사용자에게는 직관적으로 마무리된 경험을 준다. 깊은 이해는 복잡함이 아니라 명료함이다
- **점진적 정제**: 처음부터 완벽한 10을 내놓기보다, 피드백을 통해 100에서 10으로 수렴해가는 퇴고의 과정

> **Minimal for Elegant** — 인지적 소음은 최소화하되, 결과로 얻는 가치는 가장 우아하고 강력하게.

## 브랜드 성격

Sam Samuel은 **조금 더 아는 옆자리 선배**다.

경험이 있어서 팁을 줄 수 있지만, 가르치려 들지 않는다. "나도 처음에 헤맸는데"라고 말할 수 있는 사람. 사용자보다 약간 앞서 있되, 그 간격을 권위로 쓰지 않는다.

- "아, 그거? 이렇게 하면 돼요" — 아는 걸 가볍게 알려준다
- "이거 먼저 해보는 게 나을 거예요" — 경험에서 나온 순서를 제안한다
- 사용자를 초보자 취급하지 않되, 어려운 것을 쉬운 말로 설명한다
- 완성된 결과물보다 만드는 과정의 동반자 역할

> 비유하면: 카페에서 노트북 펴놓고 옆자리에서 같이 작업하는 사람. 물어보면 바로 답해주고, 안 물어보면 조용히 자기 일 한다.

## 감정적 약속

사용자가 Sam Samuel 제품을 쓴 뒤 느껴야 할 감정: **"계속 쓰고 싶다."**

화려한 첫인상이 아니라, 조용히 습관이 되는 도구. 한 번 쓰고 감탄하는 게 아니라, 매일 열게 되는 편안함. 그래서 우리 디자인의 모든 판단은 "이게 매일 쓰기에 불편하지 않은가?"로 귀결된다.

따뜻함이 가장 먼저 느껴져야 하는 순간은 **첫 사용**이다. 처음 열었을 때 막막하지 않고, 환영받는 느낌. 이 첫인상이 "계속 쓰고 싶다"로 이어지는 출발점이다. 첫 화면에서 차갑거나 복잡하면, 두 번째 방문은 없다.

## Sam Samuel이 아닌 것

브랜드가 명확하려면, 되지 않을 것도 명확해야 한다.

| 우리는 이것이 **아니다**       | 왜                                                       |
| ------------------------------ | -------------------------------------------------------- |
| 전문가 전용 도구               | 진입장벽을 높이는 순간 우리 미션과 모순된다               |
| 기능 과잉 플랫폼               | 뭐든 다 되는 도구는 아무것도 잘 못한다                    |
| 트렌디한 장난감                | 유행을 따르면 유행과 함께 사라진다                        |
| **차가운 기업 도구** (최대 경계) | 효율만 남기면 사람이 빠진다. 따뜻함이 우리의 차별점이다   |

> **가장 빠지기 쉬운 함정은 차가워지는 것이다.** 최적화하고, 정리하고, 효율을 높이다 보면 따뜻함이 사라진다. 기능을 추가할 때, UI를 다듬을 때, 문구를 쓸 때 항상 "이게 차갑지 않은가?"를 점검한다.

## 톤앤매너

### 말투: 존댓말 부드러운 체

격식체가 아니라, 부드러운 존댓말을 쓴다. "~하십시오"가 아니라 "~해보세요", "~돼요".

| 상황         | 이렇게 (O)                                    | 이렇게 안 함 (X)                               |
| ------------ | --------------------------------------------- | ---------------------------------------------- |
| 행동 유도    | "시작해볼까요? 여기를 눌러보세요."              | "시작하십시오."                                 |
| 입력 요청    | "이름만 입력하면 돼요."                         | "이름을 입력해 주십시오."                        |
| 안내         | "천천히 따라해보세요."                           | "아래 절차를 수행하세요."                        |
| 에러         | "이런, 문제가 생겼어요. 다시 해볼까요?"          | "오류가 발생했습니다. 재시도하십시오."             |

### 원칙

| 키워드       | 의미                                          | 적용                                           |
| ------------ | --------------------------------------------- | ---------------------------------------------- |
| **편안함**   | 부담 없는 말투, 부드러운 존댓말                 | 위 말투 표 참조                                 |
| **안정감**   | 급하지 않은 리듬, 여유 있는 여백               | 한 화면에 한 가지 행동만 요구                   |
| **명료함**   | 모호한 설명 없이 한 문장으로 이해              | "여기에 이름을 입력하면 프로필이 생성돼요"        |

### 글쓰기 가이드

- 짧은 문장을 쓴다. 한 문장에 한 가지 정보만 담는다.
- 기능 설명 시 "왜"를 먼저, "어떻게"를 나중에 쓴다.
- 사용자에게 입력을 요구할 때는 그 이유가 흐름 안에서 자연스럽게 드러나야 한다.
- 에러 메시지도 친근하게. 원인과 해결을 같이 알려준다.
- "~하십시오", "~바랍니다" 같은 격식체는 쓰지 않는다.
- 일반 글/Threads/회고 글감은 같은 레포 `WRITING_TECHNIQUES.md`의 퇴고 기준과 `TASTE.md` §2의 톤 기준을 따른다.
- `적/의/것/성`, `하고 있다`, `할 수 있다`, `관한`, `대한`, `있는/있어서`를 줄이고 능동형을 우선한다.
- 첫 줄은 그림이 떠오르는 한 줄 요약으로 시작하고, 뒤에 질문과 설명을 붙인다.
- 경험, 감정, 유머, 비유, 호기심을 넣어 기억에 남는 글로 만든다.

## 디자인

디자인 정본은 세 층으로 나뉜다. 이 문서는 "왜"까지만 담는다.

1. **BRAND.md** (이 문서) — 왜 존재하는가: 가치, 성격, 톤앤매너, 브랜드 상수, 캐릭터 정본
2. **[DESIGN.md](./DESIGN.md)** — 어떤 느낌이어야 하는가: 디자인 원칙, 자연의 언어, 화면 판단 테스트, 콘텐츠 비주얼(Warm Knolling)
3. **[DESIGN_SYSTEM.md](./DESIGN_SYSTEM.md)** — 어떻게 만드는가: 토큰, 컴포넌트, 제품 패턴, 모션, 접근성

### 컨셉과 판단 게이트

**미니멀하고 젠 스타일.** 100에서 10을 남기는 편집의 결과물. 타이포그래피 위주로, 기능 자체가 디자인으로 설명되어야 한다. (미니멀의 정의 — 비움이 아닌 정돈+깊이 — 상세는 DESIGN.md > 비주얼 방향)

디자인의 모든 판단은 네 가지 질문으로 귀결된다:
1. **이게 따뜻한가?** — 차가워지는 것이 가장 큰 위험이다 (안티 포지셔닝 참조)
2. **이게 자연스러운가?** — 인공적이고 기계적인 느낌을 경계한다
3. **매일 쓰기에 불편하지 않은가?** — 습관이 되는 편안함이 목표다
4. **100에서 남긴 10인가?** — 본질만 남았는지, 아니면 그냥 비어있는지 점검한다

### 캐릭터 정본: 샘(Saem)
