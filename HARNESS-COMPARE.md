# Harness Compare — GitHub AI 시스템 비교 분석

GitHub의 AI 관련 레포를 클론하여 내 시스템과 비교 분석한다.
타겟 레포만의 독특한 접근도 별도로 식별한다.

## 대상 레포 유형

| 유형 | 예시 | 비교 초점 |
|------|------|----------|
| Claude Code 하네스/설정 | 스킬, 에이전트, 훅 | 내 Claude Code 설정과 1:1 비교 |
| 멀티에이전트 오케스트레이션 | CrewAI, AutoGen, LangGraph | 내 workflow-master/OMC 오케스트레이션과 패턴 비교 |
| 자동 개선/자율 시스템 | Auto-GPT, self-improving agents | 내 infinity/agent-first 시스템과 자율성 비교 |
| AI 워크플로우/파이프라인 | LangChain, Haystack | 내 파이프라인 설계와 비교 |
| 프롬프트/설정 아카이브 | awesome-prompts, dotfiles | 구조화/관리 방식 비교 |
| 기타 AI 프로젝트 | 범용 | 설계 철학, 아키텍처 패턴 중심 비교 |

## 입력

- GitHub URL (필수) — `https://github.com/{owner}/{repo}` 형태
- 비교 관점 (선택) — 특정 영역만 비교하고 싶으면 지정

## 워크플로우

### Phase 1: 클론 및 분류

1. `/tmp/{repo}-compare`에 shallow clone (`git clone --depth 1`)
2. 파일 트리 수집 (`find -type f`, `.git` 제외)
3. **레포 유형 자동 판별** — 파일 패턴으로 분류:

| 신호 | 판별 유형 |
|------|----------|
| `SKILL.md`, `.claude/`, `agents/*.md` | Claude Code 하네스 |
| `crew.py`, `agents.yaml`, `tasks.yaml` | CrewAI 기반 오케스트레이션 |
| `autogen`, `groupchat`, `AssistantAgent` | AutoGen 기반 오케스트레이션 |
| `langgraph`, `StateGraph`, `nodes` | LangGraph 기반 워크플로우 |
| `langchain`, `chains`, `prompts` | LangChain 기반 파이프라인 |
| `setup.py`/`pyproject.toml` + agent 키워드 | Python AI 프레임워크/라이브러리 |
| `package.json` + AI 의존성 | Node.js AI 프로젝트 |
| 주로 `.md` 파일 | 프롬프트/설정 아카이브 |

4. 유형에 따라 Phase 2의 분석 전략을 분기

### Phase 2: 타겟 분석

#### 2-1. 핵심 파일 읽기

**공통 (모든 유형)**:
- `README*.md` — 개요, 설계 철학
- 진입점 파일 (main, index, app 등) — 아키텍처 파악
- 설정 파일 — 의존성, 환경 구성

**Claude Code 하네스일 때 추가**:
- `CLAUDE.md`, `settings.json`, `skills/*/SKILL.md`, `agents/*.md`

**오케스트레이션 시스템일 때 추가**:
- 에이전트 정의 파일, 태스크/워크플로우 정의, 통신/조율 로직

**자동 개선 시스템일 때 추가**:
- 피드백 루프 로직, 자기 평가/개선 메커니즘, 목표 설정 방식

> 각 파일 500줄 초과 시 첫 200줄 + 구조 요약으로 제한

#### 2-2. 설계 철학 추출
- 레포의 핵심 목적과 해결하려는 문제
- 설계 원칙 (문서 명시 + 구조에서 추론)
- 차별화 포인트 (README에서 강조하는 것)

#### 2-3. 아키텍처 분석

| 분석 축 | 추출 내용 |
|---------|----------|
| 에이전트 모델 | 에이전트 정의 방식, 역할 분리 기준, 수 |
| 오케스트레이션 | 조율 패턴 (파이프라인/팬아웃/감독자/계층적 등) |
| 통신 | 에이전트 간 데이터 전달 방식 (메시지/파일/공유상태) |
| 메모리/상태 | 상태 관리, 컨텍스트 유지, 영속성 |
| 피드백 루프 | 자동 개선, 검증, 반복 메커니즘 |
| 도구 사용 | 외부 도구 통합 방식 |
| 확장성 | 새 에이전트/도구 추가 용이성 |

#### 2-4. 특이점 탐지

다음 기준으로 "이 레포만의 독특한 접근"을 식별한다:
- **새로운 패턴**: 기존 프레임워크에서 보기 어려운 설계
- **창의적 조합**: 여러 개념을 독특하게 엮은 방식
- **문제 해결 접근**: 흔한 문제를 비상식적 방법으로 해결
- **구조적 미학**: 파일 구조, 네이밍, 추상화가 인상적인 경우
- **실전 노하우**: 문서에 녹아있는 경험 기반 교훈

### Phase 3: 내 시스템 매칭

타겟 유형에 따라 내 시스템의 대응 영역을 로드한다:

| 타겟 유형 | 내 시스템 비교 대상 |
|----------|-------------------|
| Claude Code 하네스 | `~/.claude/skills/`, `~/.claude/agents/`, `settings.json` |
| 멀티에이전트 오케스트레이션 | workflow-master, OMC 에이전트 카탈로그, 팀 파이프라인 |
| 자동 개선/자율 시스템 | infinity (agent-first), heartbeat, daily-feedback-system |
| AI 워크플로우 | CLAUDE.md 워크플로우, 스킬 체이닝 구조 |
| 프롬프트 아카이브 | prompt-archive 레포, 스킬 description 패턴 |

> Phase 2에서 발견한 비교 포인트와 관련된 부분만 선택적으로 로드한다.

### Phase 4: 비교 분석

3개 에이전트를 병렬로 실행한다:

#### Agent 1: 구조 분석가 (Explore)
- 아키텍처 레이어 비교 (추상화 수준, 모듈화)
- 파일 구조/네이밍/문서화 수준 비교
- 확장성/유지보수성 설계 비교

#### Agent 2: 패턴 분석가 (Explore)
- 오케스트레이션 패턴 비교 (조율, 통신, 상태 관리)
- 피드백 루프/자동 개선 메커니즘 비교
- 에러 핸들링/복구 전략 비교

#### Agent 3: 인사이트 추출가 (Explore)
- 타겟만의 독특한 접근 심화 분석 (최소 3개)
- 내 시스템에 없는 개념/도구/패턴 식별
- 즉시 적용 vs 장기 참고 분류 + 구체적 적용 방안

### Phase 5: 종합 보고

```markdown

|------|---|
| 유형 | {판별된 유형} |

| 기술 스택 | {주요 기술} |

### 타겟의 독특한 접근 Top 3

|---|------|------|-----------|

#### 즉시 적용 (비용 낮고 효과 큼)
```

### Phase 6: 정리

1. `/tmp/{repo}-compare` 삭제
2. 보고서 요약 출력

## 에러 핸들링

| 상황 | 대응 |
|------|------|
| 클론 실패 (네트워크) | 재시도 1회 → 실패 시 사용자에게 네트워크 확인 요청 |
| 클론 실패 (private repo) | `gh` CLI로 재시도. 그래도 실패 시 안내 |
| AI 관련 레포가 아닌 경우 | "이 레포는 AI 시스템이 아닐 수 있습니다" 안내 후 범용 구조 분석 |
| 레포가 너무 큼 (100+ 파일) | 핵심 파일만 선별. README + 진입점 + 설정 우선 |
| 모노레포 | 사용자에게 분석할 하위 경로를 물어본다 |

## 주의사항

- 타겟의 단점을 지적하지 않는다. "배울 점"과 "방향의 차이"만 구분한다
- 내 시스템이 무조건 좋다고 가정하지 않는다. 객관적으로 비교한다
- 분석 결과를 파일로 저장하지 않는다. 터미널 출력만 한다 (필요 시 사용자가 요청)
- 타겟이 Python 프레임워크면 코드를 읽고 설계를 이해한다. 실행하지는 않는다
