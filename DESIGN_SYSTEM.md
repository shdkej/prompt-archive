## 설계 원칙 (고정)

1. **선택지 최소화** — 고민을 줄인다
2. **의미 중심** — "이 색은 언제 쓰는가"가 드러나야 한다
3. **UI는 배경으로** — 사용자의 콘텐츠가 주인공
4. **컨셉이 레이아웃을 이긴다** — 제품의 본질을 가장 잘 전달하는 UI가 최선이다
5. **자연에서 가져온다** — 인공적이고 기계적인 느낌을 경계한다. 곡선, 색감, 여백 모두 자연이 기준이다

### 혁신적 UI 허용 원칙

디자인 시스템은 **제약이 아니라 도구**다.

**깰 수 있는 것:** 그리드 배치, 섹션 순서, 네비게이션 위치, 화면 전환 패턴

**반드시 지키는 것:** semantic 토큰 구조, 접근성 규칙

### UX 설계 원칙: 편집의 실천

> "100에서 10을 남기는 편집 철학"의 UX 적용. 참조: BRAND.md > 편집 철학

#### 1. 정제된 화면 구성 ('비움'이 아닌 '정제')

- 한 화면에서 사용자에게 요구하는 행동은 **하나**다
- 기능을 나열하지 않는다. 사용자의 맥락에서 **지금 필요한 것**만 보여준다
- 방대한 기능을 깊이 이해한 후 최적의 정수만 화면에 배치한다

#### 2. 모듈형 인터페이스 (사용자 주권)

- 기능을 일방적으로 제공하지 않고 **모듈로 만들어 사용자가 선택**할 수 있게 한다
- 자주 사용하는 기능은 **접근 단계를 최소화**하여 직관성을 높인다
- 단순한 패턴 매칭을 넘어, 사용자의 **의도와 맥락을 이해하고 선제적으로 제안**하는 지능형 UX를 지향한다

#### 3. 시각적 위계로 인지 부하 제거

> **가장 중요한 원칙이다.** 편집 철학, 모듈형 UX, 자연스러운 텍스처가 모두 살아있어도 시각적 위계가 무너지면 사용자는 아무것도 읽지 않는다. 다른 원칙과 충돌하면 시각적 위계를 우선한다.

- **5초 룰**: 사용자가 화면을 5초 동안 본 뒤 "가장 중요한 메시지"를 답할 수 있어야 한다. 답하지 못하면 위계는 실패다
- 사용자의 시선이 머무는 곳에 핵심 가치를 배치한다 (F-패턴, Z-패턴 활용)
- 참고 이론: 로빈 윌리엄스 4원칙(근접성·정렬·반복·대비), 게슈탈트 그룹화 원리
- **균등 배치 금지**: 모든 요소가 같은 무게를 가지면 위계가 사라진다. 1순위/2순위/3순위 정보를 명확히 구분한다
- **위계 구성 수단**: 크기, 굵기, 색 대비, 여백, 위치, 그룹핑에 질감·뎁스(원근/아웃포커싱)·움직임·리듬(크게→작게→크게)까지 **의도적으로** 조합해 "보는 순서"를 설계한다. 하나의 수단만으로는 약하다
- **설계 순서**: 컨셉 → 설계 → 구도 → 배치 → 색감
- **여백은 구조다**: 장식이 아니라 관계를 표현하는 언어다. 가까운 것은 한 그룹, 먼 것은 다른 그룹으로 읽힌다
- **주인공은 하나**: 한 화면에 강조점(primary action, hero element)은 원칙적으로 하나. 강조가 많으면 강조가 없는 것과 같다
- 어떻게 사용하는지 **디자인 자체가 설명**해야 한다 — 별도의 긴 설명이 필요 없는 직관성
- 접근성은 최소 기능이 아닌 필수 기능이다 — 모든 사용자에게 **평평한 길**을 제공한다

**리뷰 체크리스트**: UI 산출물을 리뷰할 때 "시각적 위계가 분명한가?"를 첫 번째 질문으로 삼는다. 이 질문에 통과하지 못하면 다른 디테일은 보지 않고 돌려보낸다.

#### 4. 점진적 완성 (Agile Evolution)

- 핵심 뼈대만 있는 상태에서 **사용자 피드백**을 받아 점진적으로 진화시킨다
- 작가가 글을 퇴고하듯, UX도 지속적인 **리팩토링**으로 불필요한 복잡성을 제거한다
- 빠르게 시도하여 안 되는 방법을 제거해 나가는 것이 본질(10)에 다다르는 지름길이다

#### 5. 기술적 복잡성의 명료한 환원

- 내부 로직이 아무리 복잡해도 사용자에게는 **직관적으로 마무리된 경험**을 준다
- 현실의 복잡한 문제를 **다루기 쉬운 추상화된 인터페이스**로 변환하여 제공한다
- 깊은 지식과 이해는 복잡함이 아니라 **명료함**이다

### 절대 하지 않는 것

- Atomic Design 풀세트
- blue-100 ~ blue-900 팔레트 중심 토큰
- ghost, outline, subtle, link 등 과잉 변형
- UX 패턴 대백과

## 브랜드 텍스처: Warm Tool (고정=정신, 유연=구현)

> 참조: Braun(도구의 촉감), 교보문고(따뜻한 공간감), 츠타야(큐레이션된 경험)

Sam Samuel의 시각적 서명은 **"따뜻한 도구"**다. 미니멀하지만 밋밋하지 않고, 모든 요소가 물리적 촉감을 가진다. "빨리 끝내고 나가는 도구"가 아니라 **"머물며 만드는 공간"**이어야 한다.

### 핵심 원칙 (고정)

1. 그림자에 순수 검정 금지 — 따뜻한 회색 톤으로 자연광 아래의 그림자처럼
2. 버튼은 "누를 수 있는" 물리적 피드백을 가진다
3. 입력 필드는 "오목한" 리세스 느낌을 가진다
4. 표면은 깨끗하되 그림자로 깊이감을 만든다
5. 다크모드에서도 완전 검정 금지 — 따뜻한 차콜 기반
6. 단색 배경 금지 — 미세한 명도 그라데이션으로 자연스러운 톤 변화를 준다

### 표현 방식 (유연)

아래는 **레퍼런스 구현**이다. 프로젝트 특성이나 외부 스타일 추천에 따라 변형 가능하다.

- **그림자**: 다층(2~3 layer) soft shadow 권장. 레이어 수, blur/offset 값은 자유
- **표면 처리**: gradient, inner shadow, flat 등 Warm Tool 정신이 느껴지는 방식이면 OK
- **인터랙션**: hover→떠오름, active→눌림의 촉감. 구체 구현(transform, shadow, opacity 등)은 자유

#### 배경 그래디언트 (기본 레퍼런스)

surface 토큰 3종을 조합한 그래디언트. 히어로, 랜딩 페이지 등에 기본 배경으로 사용한다. 토큰 값에 따라 자동으로 톤이 결정된다.

```css
--gradient-surface: linear-gradient(
  180deg,
  var(--color-surface) 0%,
  var(--color-bg) 50%,
  var(--color-surface-hover) 100%
);
```

### 시그니처 훅 (고정=원칙, 유연=형태)

> 스테디한 기반 + 하입한 훅 하나. 둘의 대비가 "오래 가면서도 눈에 띄는"을 만든다.

디자인 시스템(클린 화이트·토큰·Pretendard)은 **절제된 무대**다. 그 위에서 **모든 앱은 정확히 하나의 시그니처 훅**을 가진다 — 처음 본 사람이 기억하고, 다시 와도 질리지 않는 3D/모션 모먼트. 전면에 3D·모션을 흩뿌리면 화려하지만 빨리 질려서 오래 못 간다. 무대는 절제, 주연은 하나.

**원칙 (고정)**

1. **훅은 하나만** — 한 화면에 임팩트 모먼트는 1개. 흩뿌리면 지속성(오래감)과 가독성을 동시에 잃는다. 절제가 지속성을 만든다.
2. **기반은 건드리지 않는다** — 훅은 색·레이아웃·가독성을 바꾸지 않는 **인터랙션/깊이 레이어**로 얹는다. 무대(스테디)와 주연(하입)을 분리한다.
3. **깊이는 빛에서 온다** — 3D는 면을 쌓는 게 아니라 스튜디오 조명의 깊이(부유·반사·시차)로 표현한다. 과장된 베벨·네온 압출 금지.
4. **물러날 줄 안다** — `prefers-reduced-motion` 존중, 입력이 없으면 정지, 성능 예산(transform/opacity GPU 합성만) 준수. 콘텐츠 가독성이 항상 우선이다.

**형태 (유연)** — 앱 성격에 맞는 훅을 하나 고른다:

| 앱 성격 | 훅 예시 |
|---------|---------|
| 데이터/대시보드 | 마우스 추적 3D 카드 틸트 + 스튜디오 조명 반사(glare) |
| 히어로/랜딩 | 거대 타이포 시차(parallax) 깊이, 스크롤 reveal |
| 제품/갤러리 | 부유 패널(perspective float), 호버 스포트라이트 |
| 도구/유틸 | 물리감 있는 마이크로 모션(스프링, 자석 스냅) |

**레퍼런스 구현: Travel Studio Ledger** (`studio.html`) — 스튜디오 음영 무대 위에서 카드가 마우스를 따라 미세하게 기울고(3D tilt) 표면에 조명 반사가 흐른다. 거대 D-day 숫자는 시차 깊이를 가진다. 훅은 그 하나뿐, 나머지는 정적·절제다.

### WebGL 3D 훅 레시피 (고정=기법, 유연=장면)

CSS tilt·parallax로 안 되는 "진짜 입체"(지구본, 제품 뷰어, 파티클 필드)는 Three.js로 정식 구현한다. GIF·영상 루프·외부 위젯 임베드로 우회하지 않는다 — 테마도 못 넘고 데이터도 못 싣는다.

**원칙 (고정)**

1. **빌드 없이 단일 HTML** — importmap으로 three ESM을 CDN 로드. 번들러는 페이지가 앱이 될 때만 도입한다.
2. **색은 CSS 변수 단일 출처** — 3D 전용 팔레트 변수(`--g-*`)를 라이트/다크 블록에 두고, `getComputedStyle`로 읽어 머티리얼에 주입. 테마 토글이 `themechange` 이벤트를 쏘면 재주입. **텍스처 이미지는 테마를 못 넘으므로 금지** — 지오메트리+셰이더로만 그린다.
3. **실데이터가 지오메트리** — 장식용 3D가 아니라 제품 데이터를 입체로 옮긴다 (지형 → 도트, 경로 좌표 → 아크, 현재 상태 → 마커). 데이터 fetch는 본문과 promise 하나를 공유한다.
4. **깊이는 빛** — 프레넬 림 글로우(BackSide + AdditiveBlending) + 안쪽 림 셰이딩. 광원·그림자 맵 대신 뷰 각도 기반 셰이더 두 장이면 충분하다.
5. **모션 문법 4단계** — 진입(현재 상태를 향해 ease 회전) → 유휴(느린 자전) → 조작(드래그 관성, 탭은 행동) → 물러남(`prefers-reduced-motion` 정지, IntersectionObserver·`document.hidden`에서 렌더 스킵, pixelRatio ≤ 2).
6. **모바일 스크롤을 죽이지 않는다** — 풀뷰포트 캔버스는 `touch-action: pan-y`. 세로 스와이프는 페이지에게, 가로 드래그만 3D에게.
7. **실패는 조용히** — WebGL 컨텍스트 실패 시 캔버스만 안 보이고 레이아웃은 유지 (`body.globe-on` 같은 성공 클래스로 스타일 게이트).

**기법 카탈로그 (유연)** — 장면에 맞게 조합:

| 기법 | 방법 |
|------|------|
| 도트 랜드마스 | topojson 지형 → 등장방형 캔버스 래스터라이즈 → 피보나치 구면 격자 샘플 → `THREE.Points` 원형 스프라이트 셰이더 |
| 대권 아크 | 두 좌표 단위벡터 lerp→normalize, `sin(πt)`로 리프트. 지난 구간 실선 / 남은 구간 dashed |
| 화면 라벨 | 3D 좌표 `Vector3.project(camera)` → HTML 오버레이 좌표. 뒷면은 `위치·카메라 dot < 임계`로 숨김 |
| 뒷면 가림 | 도트보다 살짝 작은 불투명 구(테마 표면색)를 오클루더로 — 뒤쪽 점·선이 자연히 가려진다 |

**함정 (고정)**

- world-atlas `objects.land`는 GeometryCollection — topojson `feature()`가 FeatureCollection을 반환하므로 `features`를 순회한다.
- 경도→Y축 회전은 `rotY = -lng - 90°` (theta=lng+180 매핑 기준). 180° 틀리면 정확히 지구 반대편이 보인다 — 위치 마커가 화면 중앙에 오는지로 검증.
- `gl_PointSize`는 물리 픽셀 단위 — devicePixelRatio와 뷰포트 높이에 비례 보정하지 않으면 대형 화면에서 도트가 사라진다.
- 검증은 헤드리스 chromium(`--enable-unsafe-swiftshader`) 스크린샷으로 라이트/다크 × 데스크탑/모바일 4조합. 콘솔이 안 잡히므로 디버깅은 결과를 DOM에 쓰는 프로브 페이지 + `--dump-dom`.

**레퍼런스 구현: Travel Ledger 히어로 지구본** — `space/infra-aws-static-sites/sites/travel/dist/index.html`의 module 스크립트(본편 통합, 텍스트 좌/지구본 우 분할 구도), `dist/globe-test/index.html`(독립 실험판, 풀스크린 구도). 실제 여행 경로 31구간과 현재 도시가 지오메트리다.

### 상태 변형 원칙 (고정=패턴, 유연=구현)

모든 인터랙티브 컴포넌트는 4가지 상태를 가진다: `default` / `hover` / `active` / `disabled`

| 상태 | 느낌 | 구현은 자유 |
|------|------|------------|
| default | 안정된 표면 | gradient, flat, shadow 등 |
| hover | 떠오르는 느낌 | shadow 확대, 밝기 변화, scale 등 |
| active | 눌리는 느낌 | inner shadow, scale down, 어두워짐 등 |
| disabled | 비활성 | opacity 낮춤 (0.3~0.5 범위) |

- Form(Input/Select) 추가 상태: `focus` (집중 강조), `error` (위험 강조)
- focus glow는 중립 톤 권장 — 과도한 채색 지양

## 1. 컬러 토큰 (고정=구조, 유연=값)

### 토큰 구조 (고정)

semantic 토큰만 사용한다. 아래 토큰 이름과 역할은 고정이다.

| 토큰 | 역할 |
|------|------|
| `--color-bg` | 페이지 배경 |
| `--color-surface` | 카드/컴포넌트 배경 |
| `--color-surface-hover` | 호버 시 표면 |
| `--color-text-primary` | 주요 텍스트 |
| `--color-text-secondary` | 보조 텍스트 |
| `--color-text-muted` | 비활성 텍스트 |
| `--color-primary` | 주요 액션 (앱별 오버라이드) |
| `--color-danger` | 파괴적 액션/에러 |
| `--color-success` | 성공/완료 |
| `--color-warning` | 주의/대기 |
| `--color-border` | 경계선 |
| `--color-border-focus` | 포커스 경계선 |
| `--color-overlay` | 오버레이 배경 |

### 컬러 기조 (고정)

- **기본 모드는 라이트모드** — 밝고 따뜻한 첫인상이 브랜드의 출발점이다
- **기본은 웜 뉴트럴** — 테마가 없을 때 크림/웜그레이(브랜드 기준색 `#F0EEE9` 계열) 기반. 어떤 콘텐츠든 방해하지 않되, 차갑지 않다
- **저채도** — 고채도 네온 금지. 차분한 저채도 색조 (웜 뉴트럴 기반, 절제된 색조)
- **단색 금지** — 배경, 카드 등 넓은 면적에 순수 단색을 쓰지 않는다. 미세한 명도 그라데이션으로 자연스러운 톤 변화를 준다
- **라이트모드**: 웜 화이트/크림 기반. 순수 `#FFFFFF` 지양
- **다크모드**: 따뜻한 차콜 기반. 순수 `#000000` 금지
- **테마 적용 시**: 외부 스타일 추천(ui-ux-pro-max 등)에 따라 warm/cool/craft 등 톤 변경 가능. 단, 배경은 극단적 색상(핑크, 블루 등) 지양하고 뉴트럴 범위 내에서 조정

### 시각 표현 (유연)

variant 수, 사이즈 옵션, Compound/Flat API 선택, 시각 스타일(그림자 강도, border 유무, gradient 여부 등)은 프로젝트와 스타일 추천에 따라 자유롭게 결정한다.

### 레퍼런스 구현: Editorial Style (메인)

아래는 공식 레퍼런스 스타일이다. 새 프로젝트는 이 스타일을 기본으로 사용한다.

> 쇼케이스 파일 `sam-samuel-design-system.pen`은 현재 레포에 없다(유실). 재생성하면 여기에 다시 연결한다.

#### 컴포넌트별 다크/라이트 패턴

구체적인 hex 값은 위 "기본 팔레트"를 기준으로 한다. 아래는 **테마 전환 규칙**만 정리한다.

| 규칙 | 라이트 → 다크 |
|------|--------------|
| **primary 반전** | 어두운 bg/밝은 text ↔ 밝은 bg/어두운 text |
| **semantic 틴트** | Badge, Toast 등에 semantic 컬러의 저채도 틴트를 배경으로 사용 |

| **danger 불변** | danger 컬러는 다크/라이트 동일하게 유지 |
| **섹션 bg 교차** | 인접 섹션 간 미세한 밝기 차이로 영역 구분 |

## 5. 제품 패턴 (고정=역할/승격 기준, 유연=조합/시각 밀도)

핵심 8종 컴포넌트는 원자재다. 제품 패턴은 실제 화면에서 반복되는 표면을 조립하는 **중간 레이어**다.

새 화면을 만들 때 바로 새 컴포넌트를 만들지 않는다. 먼저 핵심 8종으로 충분한지 확인하고, 같은 조합이 두 번 이상 반복되거나 제품의 가치 판단을 담는 표면이면 제품 패턴으로 승격한다.

### 승격 기준 (고정)

아래 중 2개 이상에 해당하면 제품 패턴으로 문서화한다.

- 여러 제품 또는 화면에서 반복된다
- 사용자의 첫 행동, 저장, 승인, 복귀처럼 중요한 전환을 담당한다
- 빈 상태, 오류, 권한, 로딩처럼 경험 품질을 크게 좌우한다
- AI/자동화의 판단을 사용자가 이해하거나 거절하는 표면이다
- 운영자가 매일 쓰는 표면이라 일관된 밀도와 단축 동선이 필요하다

제품 패턴은 "컴포넌트 대백과"가 아니다. Sam Samuel 제품이 반복해서 겪는 문제를 해결하는 **작은 운영 부품표**다.

### 5.1 Character Stage / Floating HUD Shell

Sam Samuel 웹의 핵심 공간 패턴이다. 배경에는 살아 있는 3D 캐릭터 또는 3D-like scene이 있고, 버튼/카드/네비/상태 정보는 그 위에 떠 있는 HUD로 배치한다.

| 패턴 | 역할 | 필수 규칙 |
|------|------|-----------|
| **CharacterStage** | 화면의 공간적 주인공. 3D/canvas/Spline/video/image-sequence/parallax layer | 장식 배경이 아니라 제품 성격과 시스템 상태를 드러내야 함. pointer/scroll/focus/status 중 최소 1개 반응 권장 |
| **FloatingHUD** | 버튼, 카드, 상태 pill, 네비를 stage 위에 띄우는 조작층 | 읽기 대비, 포커스, 클릭 영역 보장. stage를 가리기보다 필요한 위치에 얹음 |
| **MobileStagePlacement** | 모바일 첫 화면의 안정적 배치 | 캐릭터는 작고 명확하게. 엄지 조작 영역과 핵심 CTA를 침범하지 않음. 과한 WebGL은 fallback 허용 |
| **DesktopStagePlacement** | 데스크탑의 확장 배치 | 캐릭터를 크게 풀어 쓰고 위치 제한을 줄일 수 있음. 단, 핵심 정보의 첫 5초 인지는 유지 |
| **StageFallback** | 성능/접근성/로딩 대응 | static image, pre-rendered video, image sequence, CSS parallax 등으로 stage 감각을 유지 |

구현 단계는 무조건 full 3D가 아니다. `image/video fallback → pointer parallax → lightweight 3D/Spline → full interactive Three.js` 순서로 제품 가치와 성능을 검증한다.

필수 검증:

- 모바일 390~430px에서 가로 넘침 없음
- foreground control의 터치 영역 44px 이상
- `prefers-reduced-motion`에서 stage 움직임 축소 또는 정지
- WebGL/canvas 실패 시 의미 있는 fallback 노출
- 배경 캐릭터가 텍스트 대비와 포커스를 해치지 않음

Status, Travel, Infinity처럼 시스템이 움직이는 표면은 이 패턴의 우선 적용 대상이다. 단순 문서/긴 읽기 화면은 Warm Tool 기본 배경으로 충분할 수 있다.

### 5.2 App Shell / Navigation

제품의 첫 인상과 반복 방문 동선을 담당한다.

| 패턴 | 역할 | 필수 규칙 |
|------|------|-----------|
| **PageHeader** | 화면 목적, 현재 맥락, 대표 액션 표시 | 제목 1개, 보조 설명 1개 이하, primary action 최대 1개 |
| **Topbar** | 전역 상태와 빠른 액션 | sticky 가능, 높이 고정, 아이콘 버튼은 `aria-label` 필수 |
| **Sidebar** | 데스크탑 주요 탐색 | 현재 위치 표시, 260px 기준, 보조 링크는 시각 무게 낮춤 |
| **MobileBottomNav** | 모바일 반복 탐색 | 3~5개 항목, 중앙 primary action은 한 개만 |
| **BackAction / Breadcrumb** | 현재 화면의 귀환 경로 | 사용자가 만든 콘텐츠를 잃지 않는지 먼저 확인 |

모바일 앱형 제품은 `Sidebar`를 억지로 축소하지 않는다. `MobileBottomNav` 또는 화면 상단 `BackAction`으로 재구성한다.

### 5.3 Status / Ops Surface

운영 상태, 배포 상태, 큐 상태, 작업 흐름을 빠르게 읽게 한다.

| 패턴 | 역할 | 필수 규칙 |
|------|------|-----------|
| **StatusDot** | 작은 상태 신호 | 색만으로 의미 전달 금지. label 또는 tooltip과 함께 사용 |
| **MetricPill / StatCard** | 핵심 수치 요약 | 숫자, label, 변화 방향을 분리. 장식적 큰 카드 남발 금지 |
| **HealthBadge** | 정상/주의/실패 상태 | success/warning/danger 토큰 사용, 원인 또는 다음 행동 연결 |
| **ActivityFeed / Timeline** | 최근 변화 기록 | 시간, 행위, 대상, 결과가 한 줄에서 읽혀야 함 |
| **TreeRow / HierarchyList** | surface → section → card 같은 계층 편집 | depth는 여백/선으로 표현, 액션 버튼은 hover/focus에서 정리 |

운영 화면은 조용하고 밀도 있게 만든다. 랜딩 페이지처럼 큰 히어로와 장식 카드로 풀지 않는다. 다만 Status류 핵심 웹 표면에서는 `CharacterStage / FloatingHUD`를 먼저 검토하고, 상태 카드는 stage 위에 떠 있는 조작층으로 낮춘다.

### 5.4 Data & List Interaction

찾고, 좁히고, 정렬하고, 반복 편집하는 표면이다.

| 패턴 | 역할 | 필수 규칙 |
|------|------|-----------|
| **SearchBox** | 빠른 텍스트 탐색 | placeholder는 예시 중심. 결과 없음 상태와 함께 설계 |
| **FilterBar** | 상태/카테고리 좁히기 | 필터가 4개를 넘으면 접거나 그룹화 |
| **Tabs / SegmentedControl** | 보기 전환 | 같은 데이터의 관점 전환에만 사용. 다른 작업 흐름을 숨기지 않음 |
| **SortableRow** | 순서 조정 | 키보드 대체 동작 또는 명확한 up/down 버튼 제공 |
| **DataList / CompactTable** | 반복 항목 표시 | 모바일에서는 row/card 전환. 핵심 필드 3개 이하 우선 노출 |
| **Pagination / LoadMore** | 긴 목록 분할 | 현재 범위와 남은 양을 알려줌 |

테이블은 기본값이 아니다. 비교와 스캔이 핵심일 때만 사용한다.

### 5.5 Creation Flow

작성, 미리보기, 승인, 반영 흐름을 담당한다.

| 패턴 | 역할 | 필수 규칙 |
|------|------|-----------|
| **FormSection** | 관련 입력 묶음 | Label, 설명, 오류 위치 고정 |
| **FieldHelp** | 입력 이유와 예시 제공 | 긴 설명 대신 예시 1개 우선 |
| **DraftPanel** | 임시 작성 상태 | 저장 여부, 마지막 수정 시간, 취소 경로 표시 |
| **PreviewPanel** | 반영 전 결과 확인 | 실제 공개 표면과 최대한 같은 구조 |
| **DiffPreview** | 변경 전/후 비교 | 추가/삭제/변경을 색+텍스트로 표시 |
| **InspectorSheet** | 항목 상세 편집 | 목록 맥락을 잃지 않게 side sheet 우선 |
| **ConfirmDialog** | 되돌리기 어려운 행동 확인 | 영향 범위와 취소 가능 여부를 먼저 말함 |

공개 페이지, CMS, 자동화 설정처럼 외부에 반영되는 작업은 `DraftPanel → PreviewPanel/DiffPreview → ConfirmDialog` 흐름을 기본으로 한다.

### 5.6 State Components

상태 컴포넌트는 빈 화면을 방치하지 않기 위한 품질 장치다.

| 패턴 | 역할 | 필수 규칙 |
|------|------|-----------|
| **EmptyState** | 아직 데이터가 없을 때 시작점 제공 | 왜 비었는지 + 다음 행동 1개 |
| **LoadingState / Skeleton** | 기다림 완충 | 실제 레이아웃 크기를 유지해 점프 방지 |
| **ErrorState** | 실패와 회복 경로 표시 | 사용자 탓 금지. 원인, 다시 시도, 대체 경로 제공 |
| **PermissionState** | 권한/로그인 필요 안내 | 필요한 권한과 얻는 가치를 같이 설명 |
| **Offline / DegradedState** | 일부 기능 불가 | 가능한 행동과 막힌 행동을 분리 |
| **DirtyState / UnsavedChanges** | 저장 전 이탈 방지 | 변경 있음, 저장, 버리기, 계속 편집을 명확히 표시 |

빈 상태는 "없습니다"로 끝나지 않는다. Sam Samuel의 빈 상태는 사용자를 첫 행동으로 데려가는 작은 문이다.

### 5.7 AI / Result Surface

AI와 자동화는 사용자의 판단을 대체하지 않는다. 결과 표면은 사용자가 이해하고, 고치고, 거절할 수 있어야 한다.

| 패턴 | 역할 | 필수 규칙 |
|------|------|-----------|
| **ResultCard** | AI/시스템 결과의 첫 보상 | 결론, 근거, 다음 행동을 분리 |
| **ReasonBlock** | 왜 그런 결과인지 설명 | 2~3개 근거만 노출. 긴 추론 로그 금지 |
| **LimitationNote** | 확실하지 않은 부분 표시 | 과신을 줄이되 겁주지 않음 |
| **Accept / Edit / Retry Controls** | 사용자의 최종 판단권 | 저장, 수정, 다시 시도, 그냥 닫기를 job별로 허용 |
| **SaveOrExitDecision** | 결과 후 다음 경로 | 저장이 필수가 아닌 흐름에서는 정상 종료를 보존 |

좋은 결과 카드는 "AI가 맞혔다"가 아니라 "사용자가 다음 판단을 할 수 있다"로 읽혀야 한다.

### 5.8 Content / Card-News System

콘텐츠형 산출물도 제품 경험의 일부다. 카드뉴스, Threads 초안, 라이브러리 표면은 같은 품질 기준을 따른다.

| 패턴 | 역할 | 필수 규칙 |
|------|------|-----------|
| **EditorialCard** | 인사이트 1개를 세로 카드로 전달 | hook, body, label의 위계 고정. 긴 문장은 카드 분리 |
| **CoverCard** | 세트의 첫 장 | 한 문장 주장 + 맥락 1개 |
| **StepCard** | 루틴/프레임 제시 | 3단계 우선, 각 단계는 한 줄 |
| **GalleryViewer** | 여러 카드 열람 | 1~5장 인덱스, 원본 이미지 열기, 공유 가능 |
| **ThreadDraftBlock** | 카드 세트의 본문 초안 | 구체 첫 문장, 실행 가능한 결론, 추상적 격언 금지 |

카드뉴스는 장식물이 아니라 "관찰 → 인사이트 → 적용 후보"를 보존하는 제품 출력이다.

### 5.9 Motion & Interaction Tokens

모션은 기다려주는 느낌을 만든다. 과장된 재미보다 안정적인 반응이 우선이다. 단, 사용자를 기다리게 하는 불필요한 지연은 금지.

상황별 방향:

| 상황           | 모션 방향                                                   |
| -------------- | ----------------------------------------------------------- |
| **화면 전환**  | 부드러운 fade/slide. 갑작스러운 전환 금지                    |
| **요소 등장**  | 자연스럽게 나타남. 튕기거나 과장된 바운스 없이               |
| **호버/프레스** | 미묘한 반응. 버튼이 살짝 눌리거나 카드가 살짝 떠오르는 정도  |
| **성공 피드백** | 작은 기쁨을 주는 모션. 완료 체크, 저장 확인 등               |
| **로딩**       | 기다림이 지루하지 않은 부드러운 전환                         |

| 토큰 | 레퍼런스 | 용도 |
|------|----------|------|
| `motion-fast` | 120~160ms | hover, icon button, focus |
| `motion-base` | 180~240ms | sheet, toast, row expand |
| `motion-slow` | 280~360ms | page reveal, modal entrance |
| `ease-calm` | ease-out 계열 | 일반 등장/사라짐 |
| `ease-press` | ease-in-out 계열 | button active, toggle |
| `press-scale` | 0.97~0.99 | 버튼/카드 눌림 |
| `hover-lift` | 1~3px | 클릭 가능한 표면 |

`prefers-reduced-motion`에서는 위치 이동과 scale을 줄이고 opacity 또는 instant state로 대체한다.

### 5.10 Icon / Media Rule

| 항목 | 규칙 |
|------|------|
| 아이콘 | lucide를 기본으로 사용. 직접 SVG는 브랜드/특수 도메인 자산일 때만 |
| 아이콘 버튼 | familiar symbol 우선. 낯선 아이콘은 tooltip 또는 `aria-label` 필수 |
| 크기 | dense UI 14~16px, 일반 버튼 16~18px, 큰 상태 20~24px |
| 사진 | 실제 맥락을 보여줄 때 사용. 흐릿한 분위기용 stock-like 이미지 금지 |
| 스크린샷 | 제품 판단이나 before/after를 보여줄 때 사용 |
| 빈 상태 이미지 | 장식보다 시작 예시가 우선. 일러스트는 행동 이해를 돕는 경우만 |

여행/앱 제작/카드뉴스처럼 사진과 제품 화면이 함께 움직이는 프로젝트는 사진을 배경 장식으로 쓰지 말고 판단의 증거로 쓴다.

### 5.11 Component Contract

제품 패턴을 코드로 만들 때 최소 계약을 남긴다.

```ts
type ComponentContract = {
  role: string;
  requiredProps: string[];
  variants: string[];
  states: Array<"default" | "hover" | "active" | "disabled" | "focus" | "error" | "loading">;
  accessibility: string[];
  forbidden: string[];
};
```

각 패턴 문서 또는 구현에는 아래를 포함한다.

- 역할: 이 컴포넌트가 해결하는 사용자 문제
- 필수 props: 빠지면 의미가 깨지는 데이터
- variant: 최소 집합만 허용
- states: 필요한 상태와 비어도 되는 상태
- accessibility: 키보드, aria, focus, 대비
- forbidden: 쓰면 안 되는 조합
- example: 실제 화면에 가까운 예시 1개

## 6. 마이크로카피 톤 가이드 (고정=원칙, 유연=구체 문구)

### 톤 원칙 (고정)

| 원칙 | 설명 |
|------|------|
| 동반자 톤 | 선생님이 아니라 조금 더 아는 옆자리 선배. 존댓말 부드러운 체 ("~해보세요", "~돼요") |
| 행동 유도 | 상태 설명이 아니라 다음 행동 안내 |
| 간결함 | 한 문장이면 충분 |

### 패턴 (고정)

- 버튼: 행동을 구체적으로 ("확인" → "저장하기")
- 에러: 사용자 탓 하지 않기 ("잘못된" → "확인해주세요")
- 빈 상태: 끝이 아니라 시작 (다음 행동 제안 필수)
- "~되었습니다"(수동) 대신 "~했어요"(능동)

## 7. 접근성 규칙 (전부 고정)

선택이 아니라 강제. "모든 사용자가 생산자가 된다"에서 "모든"이 진짜 모든이려면.

- 명암비: 일반 텍스트 7:1 이상 목표 (WCAG AAA)
- 컬러 독립성: 컬러만으로 의미 전달 금지. 추가 시각적 단서 제공
- 키보드: 모든 인터랙티브 요소 Tab 도달 가능, 포커스 인디케이터 명확하게
- 스크린리더 호환
- `prefers-reduced-motion` 대응 필수
- 시맨틱 마크업: Button→`<button>`, Input→`<label>` 연결, Modal→`role="dialog"` + 포커스 트랩, Toast→`role="status"`/`role="alert"`

## 9. 대시보드 레이아웃 패턴 (유연)

대시보드 구축 시 참고하는 레이아웃 패턴이다. 디자인 토큰 기반으로 다크/라이트 테마 자동 전환을 지원한다.

아래 **구조 규칙**이 정답이다. (시각 쇼케이스 `.pen` 파일은 유실 — 재생성 시 연결)

### Desktop (1440×900)

| 영역 | 규칙 |
|------|------|
| 사이드바 | 고정 260px, 로고 + 네비게이션 |
| 메인 | fill_container, 헤더(72px) + 콘텐츠(padding 32px) |
| 하단 | 차트(fill) + 활동 리스트(360px) 가로 배치 |

### Mobile (402×auto)

| 영역 | 규칙 |
|------|------|
| 사이드바 | 제거 → 하단 플로팅 탭바 전환 |
| 메트릭 카드 | 2×2 그리드 (gap 12px) |
| 차트/활동 | 세로 스택 |
| 탭바 | 플로팅 필(pill) + 상단 그라디언트 페이드 |
