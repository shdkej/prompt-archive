# 회고 리뷰 패턴

> planner가 회고 리뷰 시 참조하는 파일. 리뷰할 때마다 발견한 패턴을 누적 기록한다.
> 사용자의 글 톤, 취향, 선호/비선호를 학습하는 목적.

---

## 톤 & 취향

### 선호하는 톤
- 구체적 상황 + 판단 맥락이 드러나는 글 (신문 기사 헤드라인처럼)
- "담당자가 '메일 필요 없다'고 말한 순간 — 기능을 빼는 것이 더 나은 자동화라는 판단"

### 싫어하는 톤
- 뜬구름: "완벽히 준비된 적은 없었다 — 그냥 시작하는 게 답이었다" (자기계발서 문구)
- 작업 나열: "ZeroTier로 사내망 뚫기 성공 — 도구가 다르면 방화벽도 다르게 반응한다"
- 기술 실수 나열: "YAML 스펙 확인 없이 배포" (이건 회고가 아니라 기술 노트)

## 헤드라인 패턴

### Good (채택됨)
- "담당자가 '메일 필요 없다'고 말한 순간 — 기능을 빼는 것이 더 나은 자동화라는 판단" (03-20)
  - 왜 좋았나: 구체적 상황(담당자 피드백) + 판단(기능 제거) + 의미(더 나은 자동화)

### Bad (거부됨)
- "법무 자동화 2개(배분 + 사건사고) 피드백 반영 완료한 날 — 이력관리 워크플로우도 스킬로 정착" (03-20 원본)
  - 왜 거부됐나: 작업 완료 나열. 갈등/도전/판단이 없음
- "ZeroTier로 사내망 뚫기 성공 — 도구가 다르면 방화벽도 다르게 반응한다" (03-13)
  - 왜 거부됐나: 기술 작업 로그. 상황-판단-결과 맥락 없음
- "법무 배분 사용성 다듬고 ms-graph 우선 전략 확정 — 도구 선택이 반복 비용을 결정한다" (03-16)
  - 왜 거부됐나: 할 일 목록 + 격언 조합

## Cut 패턴

### Good
- "L-0317-1을 알면서도 가품 캡쳐에서 또 탐색부터 시작. 새 기능이라는 흥미가 검증 단계를 건너뛰게 만드는 패턴" (03-20)
  - 왜 좋았나: 기존 Logic 미실행 원인을 행동 패턴으로 짚음

### Bad
- "가품 이미지 캡쳐 자동화 시도 후 취소 → 새 기능 착수 전 실현 가능성 먼저 검증" (03-20 원본)
  - 왜 거부됐나: 일회성 기술 실수. "왜 반복하는가"가 없음
- "ZeroTier OFFLINE 삽질 — privileged: true가 securityContext 밖에 있어서 무시됨" (03-13)
  - 왜 거부됐나: 기술 디버깅 로그

---

## Core 패턴

### Good
- "법무 배분 담당자 피드백 '메일은 안 본다' → 메일 발송 제거, 채널+플래너 알림 중심으로 전환. 사용자가 실제로 보는 채널에 집중하는 것이 자동화 체감 품질을 높인다는 판단" (03-20)
  - 왜 좋았나: 상황(피드백) → 판단(제거) → 근거(체감 품질)

### Bad
- "법무 자동화 2건 모두 피드백을 당일 반영 완료. 세션관리 모드로 작업 상태를 추적하면서 빠르게 전환" (03-20 원본)
  - 왜 거부됐나: 속도 자랑. 판단 기록이 아님
- "네트워크 보안 구조 먼저 확정하니 구현이 수월했음" (03-13)
  - 왜 거부됐나: 작업 후기

---

## Logic 패턴

### Good
- "자동화 개선 = 기능 추가만이 아니라, 안 쓰는 기능을 빼는 것도 포함. 메일 제거 후 흐름이 오히려 깔끔해짐 L-0320-1 (Less is More)" (03-20)
  - 왜 좋았나: 구체적 경험에서 나온 규칙, 새로운 관점

### Bad
- "k8s manifest 변경 시 helm template으로 렌더링 검증" (03-13)
  - 왜 거부됐나: 기술 규칙. 삶의 규칙이 아님
- 같은 교훈을 새 번호로 반복 생성 (03-17 L-0317-1 ≈ 03-20 원본 L-0320-1)
  - 왜 거부됐나: 중복. 미실행 원인을 Cut에서 다뤄야 함

---

## 메타 규칙

- 같은 교훈이 반복되면 Logic을 새로 만들지 말고, Cut에서 "왜 이미 배운 걸 또 못 지켰는지" 파기
- 매일 억지로 Logic을 만들지 않기. 없으면 "새로운 Logic 없음"
- 기술 디테일은 회고에 넣지 않기. 회고는 상황-판단-결과의 맥락
- "Antifragile/Leverage" 같은 태그보다 구체적 라벨 선호 (Less is More, Fail-fast 등)

## 2026-04-17 학습

### Headline Good
- `"동작이 기대와 너무 달라서 새로 하는 게 낫다" — koco-tester-manage를 버리고 invitation을 Playwright로 다시 지은 날` (04-17 최종)
  - 왜 좋았나: 한 축(폐기 판단)으로 집중, 사용자 발화가 그대로 들어가 맥락 선명

### Headline Bad
- `"원하는 대로 안 동작"이라는 한 마디에 koco-tester-manage를 버리기로 한 날 — 고쳐 쓰기의 손익분기점` (planner 1차안)
  - 왜 거부됐나: "손익분기점"이 경영서 톤. 사용자 실제 표현("동작이 기대와 너무 달라서 새로 하는게 낫다")을 쓰는 편이 진솔

### Cut Good
- 04-17: "초기엔 '차트가 audit을 그리고 있다'고 가정부터 잡고 코드를 뒤짐. 실제로는 날짜 필드 한 줄의 타임존 문제였음. 표면 증상을 데이터로 반증하지 않고 구조부터 의심하는 패턴"
  - 왜 좋았나: 구체 가설(audit 차트) + 실제 원인(타임존) 대비로 반복 패턴을 정확히 짚음

### Cut Bad
- 04-17 초안: "표면 증상에 속지 않고 원인까지 추적"
  - 왜 거부됐나: 자기 칭찬 문구. Cut은 "반복되는 실수"를 짚는 자리

### Core 작성 주의
- 과도한 해석 금지 — 04-17에서 "헬퍼 미러링 자체가 재발 위험"까지 덧붙였더니 사용자가 "무슨말인지 모르겠음"으로 반려. 관찰·판단까지만 쓰고 파생 리스크는 본인 확인 없이 추가 금지

### Logic Good
- `L-0417-1`: "동작이 기대와 너무 다른 자동화는 고치기보다 버리고 새로 만든다. 진단 비용이 재작성 비용을 넘는 지점이 있다"
  - 왜 좋았나: L-0320-1(Less is More)와 겹치지 않는 "교체 임계"라는 새 판단 기준. 사용자 실제 발화가 근거로 붙음
